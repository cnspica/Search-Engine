            <div class="post-text" itemprop="text"> 
             <p>When (accidentally) clicking "mobile" in the footer, one gets the mobile site in a regular browser too. To switch back, one can click "full site" in that same footer:</p> 
             <p><img src="http://i.stack.imgur.com/O4yzt.png" alt="footer on mobile site"></p> 
             <p>However: when hovering it, the mouse is changed as to select text, rather than to indicate one can click (even though it's clickable).</p> 
             <p>All other links do change the mouse into a pointer.</p> 
             <p>(Not a big deal, except when somehow <a href="http://meta.stackexchange.com/questions/108039/why-it-is-showing-me-mobile-version-of-stackoverflow-com-in-google-chrome">accidentally</a> getting into the mobile mode, I guess.)</p>

            <div class="post-text" itemprop="text"> 
             <p>As I noted earlier in the comments, this should be fixable with one line of CSS:</p> 
             <pre class="lang-css prettyprint-override"><code>a[onclick] { cursor: pointer }
</code></pre> 
             <p>As far as I can tell, this fixes the issue (and any possible similar issues elsewhere in the user interface) with no harmful side effects.</p> 
             <p>Since the SE folks don't, alas, seem very interested in fixing this, I've added the CSS rule above to the <a href="http://stackapps.com/questions/4486/stack-overflow-unofficial-patch">Stack Overflow Unofficial Patch</a>, a collection of client-side CSS and JavaScript fixes packaged as a GreaseMonkey-style user script. If you're using a compatible browser (Firefox, Chrome, Opera, possibly Safari and others, but unfortunately not IE or most mobile browsers), please give it a try!</p> 
             <p><strong>Update:</strong> In the upcoming release of SOUP (v1.17.devel) this rule has been generalized to:</p> 
             <pre class="lang-css prettyprint-override"><code>a[onclick], a:not([name]) { cursor: pointer }
</code></pre> 
             <p>This rule catches some extra pointer-less pseudo-links that the earlier one didn't, such a comment delete links, and should hopefully still remain free of false matches.</p>

             <p>Inspecting the footer links with Google Chrome, I notice that the other links use the <code>cursor: auto</code> property, while the "full site" link doesn't have that CSS property.</p> 
             <p>Actually, the "full site" link is merely an <code>&lt;a&gt;</code> element, without any "href" attribute, and with an "onclick" event that switches the format with a call to <code>StackExchange.switchMobile()</code>. For that link, the cursor should be explicitly set.</p> 
             <pre><code>&lt;div class="footer-links"&gt;
   &lt;a href="http://chat.stackoverflow.com"&gt;chat&lt;/a&gt;
   ...
   &lt;a href="mailto:team@stackoverflow.com"&gt;contact us&lt;/a&gt;
   &lt;a onclick="StackExchange.switchMobile('off', '/?__=...')"&gt;full site&lt;/a&gt;
&lt;/div&gt;
</code></pre> 
             <p><img src="http://i.stack.imgur.com/x8yyJ.png" alt="footer link"></p> 
             <p><img src="http://i.stack.imgur.com/bDBJx.png" alt="&quot;full site&quot; link"></p>

            <div class="post-text" itemprop="text"> 
             <p>Mobile devices that get the theme don't <em>have</em> hover, so we intentionally put little to no effort to support it...to instead reduce the load time as much as possible. Many areas don't have hover that traditionally would if you poke around.</p> 
             <p>Also, we have to use a JavaScript link (and cannot use a simple GET URL), to <a href="http://meta.stackexchange.com/questions/98925/serious-xsrf-bug-in-the-new-mobile-theme">avoid abuse</a>. Not all links should have the <code>cursor: pointer</code> style, as it can cause mobile issues, so the no-cursor is intentional.</p>

           <td class="answercell"> 
            <div class="post-text" itemprop="text"> 
             <p>I've noticed this before (when inspecting the mobile site on a desktop)</p> 
             <p>I don't think that the cursor is a big issue. Yes, it will show up on accidental switches to mobile, and I guess it makes sense to use <code>cursor:pointer</code> if it's being explicitly set.</p> 
             <p>What is an issue is that the mobile site is locked in with Javascript. In other words, you cannot <em>leave</em> the mobile site if Javascript is disabled. Thr same goes for joining the mobile site. I always use JS, so I don't care much about this, but it occurred to me that there are older phones that support a half baked version of JS. And switching to/from the mobile site is pretty important if your phone cannot handle the full site.</p> 
             <p>So, I ask the devs, what's wrong with <code>href</code>? Nothing to be lost by using it, and it's a tiny change.</p>
