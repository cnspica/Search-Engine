            <div class="post-text" itemprop="text"> 
             <p>I encountered the same problem and found that if I issue following command in the interactive shell, the virtual machine would boot into Ubuntu:</p> 
             <pre><code>fs0:\efi\ubuntu\grubx64.efi
</code></pre> 
             <p>(Use backslash, forward slash does not work. Commands in UEFI interactive shell are case insensitive.)</p> 
             <p>My VirtualBox version is 4.3.20 r96997, Ubuntu version is 14.10 AMD64. I do not know why this happens and how to solve it. Just found this not elegant and still a little bit troublesome workaround.</p> 
             <p>Update 1:</p> 
             <p>I read <a href="http://askubuntu.com/a/542734/368660">this</a>, tracked down to <a href="https://bugs.launchpad.net/ubuntu/+source/efibootmgr/+bug/1363719">a bug report</a> and found <a href="https://bugs.launchpad.net/ubuntu/+source/efibootmgr/+bug/1363719/comments/4">a better workaround</a>.</p> 
             <p>Update 2:</p> 
             <p>Workaround in Update 1 failed. I turned off the virtual machine, launched it. And it booted into UEFI Interactive shell again. According to <a href="http://bbs.archlinux.org/viewtopic.php?pid=1231228#p1231228">this</a>, the problem was probably cause by a VirtualBox bug. I am still looking for further solution to this.</p> 
             <p>Update 3:</p> 
             <p>Finally found an solution. According to <a href="http://forums.virtualbox.org/viewtopic.php?f=3&amp;t=61970#p289297">this</a>, you need to create a startup.nsh script manually. Except for the method in mentioned post, you can also do this:</p> 
             <pre><code>$ sudo mount /dev/sda1 /mnt
$ cd /mnt
$ sudo sh -c "echo '\EFI\ubuntu\grubx64.efi' &gt; startup.nsh"
</code></pre>

             <p>I had the same problem (with the <strong>EFI enabled</strong> because I couldn't get it to run otherwise). Strange; although installation of Ubuntu 14.04.2 got no error, installing Kubuntu 15.04 failed to finalize. It froze at the very end when asked to remove DVD. After reset, it booted fine but after power down it brought up shell.</p> 
             <p>So, to avoid shell type:</p> 
             <blockquote> 
              <p>fs0:<br> edit startup.nsh </p> 
             </blockquote> 
             <p>and on the opened window add these 2 lines:</p> 
             <blockquote> 
              <p>FS0:<br> \EFI\ubuntu\grubx64.efi</p> 
             </blockquote> 
             <p>press <kbd>Ctrl</kbd> + <kbd>s</kbd> and <kbd>Enter</kbd> to save and <kbd>Ctrl</kbd> + <kbd>q</kbd> to quit. Then restart VM.</p> 
             <p>Alternatively, you can always use these 2 lines to exit shell and boot OS. But second time you restart you will be in a shell again and to avoid it edit <code>startup.nsh</code>.</p>

             <input type="hidden" name="_id_" value="744903"> 
             <a class="vote-up-off" title="This answer is useful">up vote</a>

           <td class="answercell"> 
            <div class="post-text" itemprop="text"> 
             <p>After installing Kubuntu15.10 in VirtualBox5 with UEFI, the reboot of the VM fails.</p>
             <p> Adding the line </p>
             <pre>FS0:\EFI\ubuntu\grubx64.efi</pre> in the UEFI-Shell doesn't help.
             <p></p> 
             <p>And the solution with creating a new startup.nsh file in Kubuntu15.10 in the chroot environment with </p>
             <pre>sudo echo '\EFI\ubuntu\grubx64.efi' &gt; startup.nsh </pre> gives also no improvement.
             <p></p> 
             <p>I found the solution:</p>
             <p> The problem was, that the directory /boot/efi/EFI/ubuntu with the file grubx64.efi doesn't exist.</p> 
             <p>After booting a Live-CD and changing to the chroot environment, I installed the missing packets and create the needed NVRAM entry with:<br></p> 
             <pre>sudo apt-get install grub-efi-amd64-signed shim-signed
sudo update-grub</pre>
