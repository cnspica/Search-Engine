            <div class="post-text" itemprop="text"> 
             <p>In some Bourne-like shells, the <code>read</code> builtin can not read the whole line from file in <code>/proc</code> (the command below should be run in <code>zsh</code>, replace <code>$=shell</code> with <code>$shell</code> with other shells):</p> 
             <pre><code>$ for shell in bash dash ksh mksh yash zsh schily-sh heirloom-sh "busybox sh"; do
  printf '[%s]\n' "$shell"
  $=shell -c 'IFS= read x &lt;/proc/sys/fs/file-max; echo "$x"'       
done
[bash]
602160
[dash]
6
[ksh]
602160
[mksh]
6
[yash]
6
[zsh]
6
[schily-sh]
602160
[heirloom-sh]
602160
[busybox sh]
6
</code></pre> 
             <p><a href="http://pubs.opengroup.org/onlinepubs/9699919799/utilities/read.html#tag_20_109_06" rel="nofollow"><code>read</code> standard requires the standard input need to be a text file</a>, does that requirement cause the varied behaviors?</p> 
             <hr> 
             <p>Read the POSIX definition of <a href="http://pubs.opengroup.org/onlinepubs/9699919799/basedefs/V1_chap03.html#tag_03_397" rel="nofollow">text file</a>, I do some verification:</p> 
             <pre><code>$ od -t a &lt;/proc/sys/fs/file-max 
0000000   6   0   2   1   6   0  nl
0000007

$ find /proc/sys/fs -type f -name 'file-max'
/proc/sys/fs/file-max
</code></pre> 
             <p>There's no <code>NUL</code> character in content of <code>/proc/sys/fs/file-max</code>, and also <code>find</code> reported it as a regular file (Is this a bug in <code>find</code>?).</p> 
             <p>I guess the shell did something under the hood, like <code>file</code>:</p> 
             <pre><code>$ file /proc/sys/fs/file-max
/proc/sys/fs/file-max: empty
</code></pre>

           <td class="answercell"> 
            <div class="post-text" itemprop="text"> 
             <p>The problem is that those <code>/proc</code> files on Linux appear as text files as far as <code>stat()/fstat()</code> is concerned, but do not behave as such.</p> 
             <p>Because it's dynamic data, you can only do one <code>read()</code> system call on them (for some of them at least). Doing more than one could get you two chunks of two different contents, so instead it seems a second <code>read()</code> on them just returns nothing (meaning end-of-file) (unless you <code>lseek()</code> back to the beginning (and to the beginning only)).</p> 
             <p>The <code>read</code> utility needs to read the content of files one byte at a time to be sure not to read past the newline character. That's what <code>dash</code> does:</p> 
             <pre><code> $&nbsp;strace -fe read dash -c 'read a &lt; /proc/sys/fs/file-max'
 read(0, "1", 1)                         = 1
 read(0, "", 1)                          = 0
</code></pre> 
             <p>Some shells like <code>bash</code> have an optimisation to avoid having to do so many <code>read()</code> system calls. They first check whether the file is seekable, and if so, read in chunks as then they know they can put the cursor back just after the newline if they've read past it:</p> 
             <pre><code>$&nbsp;strace -e lseek,read bash -c 'read a' &lt; /proc/sys/fs/file-max
lseek(0, 0, SEEK_CUR)                   = 0
read(0, "1628689\n", 128)               = 8
</code></pre> 
             <p>With <code>bash</code>, you'd still have problems for proc files that are more than 128 bytes large and can only be read in one read system call.</p> 
             <p><code>bash</code> also seems to disable that optimization when the <code>-d</code> option is used.</p> 
             <p><code>ksh93</code> takes the optimisation even further so much as to become bogus. ksh93's <code>read</code> does seek back, but remembers the extra data it has read for the next <code>read</code>, so the next <code>read</code> (or any of its other builtins that read data like <code>cat</code> or <code>head</code>) doesn't even try to <code>read</code> the data (even if that data has been modified by other commands in between):</p> 
             <pre><code>$&nbsp;seq 10 &gt; a; ksh -c 'read a; echo test &gt; a; read b; echo "$a $b"' &lt; a
1 2
$&nbsp;seq 10 &gt; a; sh -c 'read a; echo test &gt; a; read b; echo "$a $b"' &lt; a
1 st
</code></pre>

            <div class="post-text" itemprop="text"> 
             <p>If you are interested in knowing <strong>why?</strong> this is so, you can see the answer in the kernel sources <a href="http://lxr.free-electrons.com/source/kernel/sysctl.c#L2344" rel="nofollow">here</a>:</p> 
             <pre><code>    if (!data || !table-&gt;maxlen || !*lenp || (*ppos &amp;&amp; !write)) {
            *lenp = 0;
            return 0;
    }
</code></pre> 
             <p>Basically, seeking (<code>*ppos</code> not 0) is not implemented for reads (<code>!write</code>) of sysctl values that are numbers. Whenever a read is done from <code>/proc/sys/fs/file-max</code>, the routine in question <code>__do_proc_doulongvec_minmax()</code> is called from the entry for <code>file-max</code> in the configuration <a href="http://lxr.free-electrons.com/source/kernel/sysctl.c#L1619" rel="nofollow">table</a> in the same file.</p> 
             <p>Other entries, such as <code>/proc/sys/kernel/poweroff_cmd</code> are implemented via <code>proc_dostring()</code> which does allow seeks, so you can do <code>dd bs=1</code> on it and read from your shell with no problems.</p> 
             <p>Note that since kernel 2.6 most <code>/proc</code> reads were implemented via a new API called <a href="http://lxr.free-electrons.com/source/Documentation/filesystems/seq_file.txt" rel="nofollow">seq_file</a> and this supports seeks so eg reading <code>/proc/stat</code> should not cause problems. The <code>/proc/sys/</code> implementation, as we can see, does not use this api.</p>

           <td class="answercell"> 
            <div class="post-text" itemprop="text"> 
             <p>On the first attempt, this looks like a bug in the shells that return less than a real Bourne Shell or its derivatives return (sh, bosh, ksh, heirloom).</p> 
             <p>The original Bourne Shell tries to read a block (64 bytes) newer Bourne Shell variants read 128 bytes, but they start reading again if there is no new line character.</p> 
             <p>Background: /procfs and similar implementations (e.g. the mounted <code>/etc/mtab</code> virtual file) have dynamic content and a <code>stat()</code> call does not cause the <strong>re-creation</strong> of the dynamic content first. For this reason, the size of such a file (from reading until EOF) may differ from what <code>stat()</code> returns.</p> 
             <p>Given that the POSIX standard requires utilities to expect <em>short reads</em> at any time, software that believes that a <code>read()</code> that returns less than the <em>ordered</em> amount of bytes is an EOF indication are broken. A correctly implemented utility calls <code>read()</code> a second time in case that it returns less than expected - until a 0 is returned. In case of the <code>read</code> builtin, it would of course be sufficient to read until <code>EOF</code> <strong>or</strong> until a <code>NL</code> is seen.</p> 
             <p>If you run <code>truss</code> or a truss clone, you should be able to verify that incorrect behavior for the shells that only return <code>6</code> in your experiment.</p> 
             <p>In this special case, it seems to be a Linux kernel bug, see:</p> 
             <pre><code>$ sdd -debug bs=1 if= /proc/sys/fs/file-max 
Simple copy ...
readbuf  (3, 12AC000, 1) = 1
writebuf (1, 12AC000, 1)
8readbuf  (3, 12AC000, 1) = 0

sdd: Read  1 records + 0 bytes (total of 1 bytes = 0.00k).
sdd: Wrote 1 records + 0 bytes (total of 1 bytes = 0.00k).
</code></pre> 
             <p>The Linux kernel returns <strong>0</strong> with the second <code>read</code> and this is of course incorrect.</p> 
             <p>Conclusion: Shells that first try to read a large enough chunk of data do not trigger this Linux kernel bug.</p>
